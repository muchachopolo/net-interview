Section 01

C# Language Fundamentals

Delegates - Under the Hood

A delegate is a type-safe function pointer. Under the hood, when you declare a delegate, the C# compiler generates a class that inherits from System.MulticastDelegate. This class contains an invocation list (a linked list of method references).

How Delegates Work Internally

When you create a delegate instance, the compiler creates an object with: Target (the object instance for instance methods, null for static), Method (a MethodInfo reference), and an invocation list for multicast delegates.

// Delegate declaration - compiler generates a class inheriting MulticastDelegate
public delegate int MathOperation(int x, int y);

// Built-in delegate types
Func<int, int, int> add = (x, y) => x + y;        // Returns value
Action<string> log = msg => Console.WriteLine(msg); // No return (void)
Predicate<int> isEven = n => n % 2 == 0;            // Returns bool

// Multicast delegate - invocation list
Action<string> handler = msg => Console.WriteLine("Handler 1: " + msg);
handler += msg => Console.WriteLine("Handler 2: " + msg);
handler("Hello"); // Both handlers execute in order

// What the compiler generates (simplified):
public sealed class MathOperation : MulticastDelegate
{
    public MathOperation(object target, IntPtr method) { }
    public int Invoke(int x, int y) { }
    public IAsyncResult BeginInvoke(int x, int y, AsyncCallback cb, object state) { }
    public int EndInvoke(IAsyncResult result) { }
}

Events and Event Patterns

Events are a restricted form of delegates. The event keyword restricts external code to only += and -= operations, preventing assignment or direct invocation from outside the class.

public class DrugPriceService
{
    // Event declaration - only this class can invoke it
    public event EventHandler<PriceChangedEventArgs> PriceChanged;

    protected virtual void OnPriceChanged(PriceChangedEventArgs e)
    {
        // Thread-safe invocation pattern
        PriceChanged?.Invoke(this, e);
    }

    public void UpdatePrice(string drugId, decimal newPrice)
    {
        // business logic...
        OnPriceChanged(new PriceChangedEventArgs(drugId, newPrice));
    }
}

public class PriceChangedEventArgs : EventArgs
{
    public string DrugId { get; }
    public decimal NewPrice { get; }
    public PriceChangedEventArgs(string drugId, decimal price)
        => (DrugId, NewPrice) = (drugId, price);
}

Lambda Expressions and Closures

Closure Variable Capture - Memory Implications

Closures capture variables, not values. The compiler generates a hidden class to hold captured variables, which extends their lifetime. This can cause memory leaks if closures hold references to large objects or create unintended long-lived references.

// Closure captures the VARIABLE, not the value
var actions = new List<Action>();
for (int i = 0; i < 5; i++)
{
    actions.Add(() => Console.WriteLine(i)); // All print 5!
}

// Fix: capture a local copy
for (int i = 0; i < 5; i++)
{
    int captured = i;
    actions.Add(() => Console.WriteLine(captured)); // Prints 0,1,2,3,4
}

// What the compiler generates for a closure:
private sealed class <>c__DisplayClass
{
    public int i;
    public void <Method>b__0() => Console.WriteLine(i);
}

// Memory leak example with closures
public class DataProcessor
{
    private byte[] _largeBuffer = new byte[10_000_000];

    public Func<int> GetCounter()
    {
        int count = 0;
        // This closure captures 'this', keeping _largeBuffer alive!
        return () => ++count;
    }
}

Section 02

CLR and Garbage Collection Internals

Role of the CLR

The Common Language Runtime manages code execution, providing: memory management (GC), JIT compilation (IL to native code), type safety, exception handling, and thread management.

JIT Compilation Flow

C# source → Roslyn compiler → IL (Intermediate Language) in assembly → CLR loads assembly → JIT compiles methods on first call to native code → Subsequent calls use cached native code. Tiered compilation (.NET Core+) re-JITs hot methods with better optimizations.

Garbage Collection Generations

// GC Generational Model
// Gen 0: Short-lived objects (most objects die here). ~256KB - very fast collection
// Gen 1: Buffer between short and long-lived. Objects that survive Gen 0.
// Gen 2: Long-lived objects (singletons, caches, static data). Full GC - expensive!

// Large Object Heap (LOH): Objects >= 85,000 bytes go here directly
// LOH is collected with Gen 2 but NOT compacted by default (fragmentation!)

// Force compaction (rarely needed):
GCSettings.LargeObjectHeapCompactionMode = GCLargeObjectHeapCompactionMode.CompactOnce;
GC.Collect();

// GC Modes:
// Workstation GC: concurrent, lower latency, single GC thread
// Server GC: one GC thread per core, higher throughput, higher memory usage

// Configure in .csproj or runtimeconfig.json:
// <ServerGarbageCollection>true</ServerGarbageCollection>

IDisposable Pattern

public class DatabaseConnection : IDisposable
{
    private NpgsqlConnection _connection;  // Managed resource
    private IntPtr _nativeHandle;          // Unmanaged resource
    private bool _disposed = false;

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this); // Prevent finalizer from running
    }

    protected virtual void Dispose(bool disposing)
    {
        if (!_disposed)
        {
            if (disposing)
            {
                // Dispose managed resources
                _connection?.Dispose();
            }
            // Always release unmanaged resources
            if (_nativeHandle != IntPtr.Zero)
            {
                CloseHandle(_nativeHandle);
                _nativeHandle = IntPtr.Zero;
            }
            _disposed = true;
        }
    }

    // Finalizer - safety net for unmanaged resources
    ~DatabaseConnection() => Dispose(false);
}

// Proper usage with using statement
await using var conn = new NpgsqlConnection(connectionString);
// Dispose called automatically at end of scope
Common Memory Leaks in Managed Code
  • Event handlers not unsubscribed: Publisher holds reference to subscriber via delegate
  • Static references: Static collections growing without bounds
  • Closures capturing 'this': Lambda in long-lived callback holds entire object
  • Unmanaged handles: P/Invoke resources without proper Dispose
  • Timer callbacks: System.Timers.Timer holds reference to callback target
  • DbContext not disposed: Change tracker accumulates entities

Finalizers vs Dispose

Performance No overhead Objects promoted to Gen 1+ (survive longer)

Async/Await Deep Dive

What async/await Actually Does

The compiler transforms an async method into a state machine. Each await becomes a state transition. The state machine tracks where to resume after the awaited task completes.

// What you write:
public async Task<DrugPrice> GetPriceAsync(string drugId)
{
    var drug = await _repository.FindAsync(drugId);   // State 0 -> 1
    var price = await _pricingService.CalculateAsync(drug); // State 1 -> 2
    return price;
}

// What the compiler generates (simplified):
private struct <GetPriceAsync>d__0 : IAsyncStateMachine
{
    public int <>1__state;           // Current state
    public AsyncTaskMethodBuilder<DrugPrice> <>t__builder;
    private TaskAwaiter<Drug> <>u__1;  // Awaiter for first await

    public void MoveNext()
    {
        switch (<>1__state)
        {
            case 0: /* Start first await */ break;
            case 1: /* Resume after first, start second */ break;
            case 2: /* Resume after second, return */ break;
        }
    }
}

I/O-bound vs CPU-bound

// I/O-bound: use async/await directly (no thread needed during I/O wait)
public async Task<string> GetDataAsync()
{
    using var client = new HttpClient();
    return await client.GetStringAsync("https://api.example.com/data");
    // Thread is FREE during the HTTP call - returned to pool
}

// CPU-bound: use Task.Run to offload to ThreadPool
public async Task<decimal> CalculateComplexPriceAsync(Drug[] drugs)
{
    return await Task.Run(() =>
    {
        // Heavy computation - runs on ThreadPool thread
        return drugs.Sum(d => ComputePrice(d));
    });
}

Deadlocks and SynchronizationContext

Classic Deadlock Pattern

In ASP.NET Framework (not Core), calling .Result or .Wait() on an async method blocks the SynchronizationContext thread. When the async method tries to resume on that same context, it deadlocks. ASP.NET Core doesn't have a SynchronizationContext, but you should still avoid blocking.

// DEADLOCK in ASP.NET Framework (not Core)
public ActionResult GetPrice()
{
    // Blocks the request thread
    var result = GetPriceAsync().Result; // DEADLOCK!
    return Ok(result);
}

// FIX 1: Use async all the way
public async Task<ActionResult> GetPrice()
{
    var result = await GetPriceAsync();
    return Ok(result);
}

// FIX 2: ConfigureAwait(false) in library code
public async Task<decimal> GetPriceAsync(string id)
{
    var data = await _db.FindAsync(id).ConfigureAwait(false);
    // Doesn't try to resume on original context
    return data.Price;
}

// async void - ONLY for event handlers! Exceptions crash the process.
async void Button_Click(object sender, EventArgs e) { } // OK - event handler
async void DoWork() { } // BAD - exceptions unobserved, can't await

// When Task.Run is appropriate:
// - CPU-bound work in an async context (don't block the calling thread)
// - NEVER in ASP.NET for I/O work (wastes a thread for no benefit)

// IQueryable - builds expression tree, translates to SQL
IQueryable<Drug> query = _context.Drugs
    .Where(d => d.Price > 100)     // Expression tree - not executed yet
    .OrderBy(d => d.Name)          // Still building the expression
    .Take(10);                     // Still not executed

var results = await query.ToListAsync(); // NOW executes: SELECT TOP 10 ... WHERE Price > 100 ORDER BY Name

// DANGER: Converting to IEnumerable too early
var allDrugs = _context.Drugs.AsEnumerable() // Loads ALL rows into memory!
    .Where(d => d.Price > 100)                // Filters in memory - wasteful
    .ToList();

Expression<Func<Drug, bool>> expr = d => d.Price > 100;
// This is DATA (a tree structure), not code
// EF Core's LINQ provider walks this tree and generates SQL

Deferred Execution

// Deferred: query not executed until enumerated
var query = drugs.Where(d => d.IsActive); // No execution yet

drugs.Add(new Drug { IsActive = true }); // Add after query definition

foreach (var d in query) { } // Executes NOW - includes the new drug!

// Immediate execution operators: ToList(), ToArray(), Count(), First(), Sum()
// These force evaluation immediately

// Multiple enumeration problem
IEnumerable<Drug> expensive = GetDrugs().Where(d => d.Price > 1000);
var count = expensive.Count();     // Enumerates once
var first = expensive.First();     // Enumerates AGAIN from start
// Fix: materialize with .ToList() if you need multiple passes

Common Performance Pitfalls

N+1 Query Problem with LINQ

Accessing navigation properties inside a loop without eager loading generates one query per iteration. Use .Include() or projection to avoid this.

// N+1 Problem
var orders = await _context.Orders.ToListAsync(); // 1 query
foreach (var order in orders)
{
    Console.WriteLine(order.Customer.Name); // N queries! (lazy loading)
}

// Fix: Eager loading
var orders = await _context.Orders
    .Include(o => o.Customer) // JOIN in single query
    .ToListAsync();

// Fix: Projection (even better - only select what you need)
var orderDtos = await _context.Orders
    .Select(o => new { o.Id, CustomerName = o.Customer.Name })
    .ToListAsync();
Section 05

EF Core - Tracking and Performance

DbContext Lifecycle

DbContext Scoping

DbContext is designed as a short-lived unit of work. In ASP.NET Core, it's registered as Scoped (one per HTTP request). It contains the change tracker, manages database connections, and caches entity metadata. Never use a singleton DbContext in a web app.

// Registration - Scoped by default
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(connectionString)
           .EnableSensitiveDataLogging() // Dev only - shows param values in logs
           .LogTo(Console.WriteLine, LogLevel.Information));

// DbContext lifecycle per request:
// 1. Request arrives -> scope created -> DbContext instantiated
// 2. Queries execute, entities tracked
// 3. SaveChanges() persists all tracked changes
// 4. Request ends -> scope disposed -> DbContext disposed

Change Tracking

// EF Core tracks entity states:
// Added, Modified, Deleted, Unchanged, Detached

var drug = await _context.Drugs.FindAsync(id); // Tracked (Unchanged)
drug.Price = 99.99m;                            // Auto-detected (Modified)
await _context.SaveChangesAsync();              // UPDATE only Price column

// AsNoTracking - for read-only queries (faster, less memory)
var drugs = await _context.Drugs
    .AsNoTracking()              // No change tracking overhead
    .Where(d => d.IsActive)
    .ToListAsync();

// When to use AsNoTracking:
// - Read-only pages / API responses
// - Reports and dashboards
// - When you won't modify the entities

// Bulk operations - avoid tracking overhead
await _context.Drugs
    .Where(d => d.ExpiryDate < DateTime.UtcNow)
    .ExecuteUpdateAsync(s => s.SetProperty(d => d.IsActive, false));
// Direct SQL UPDATE - no entities loaded into memory

Loading Strategies

// Eager Loading - Include related data in initial query
var orders = await _context.Orders
    .Include(o => o.Items)
        .ThenInclude(i => i.Drug)
    .Include(o => o.Customer)
    .ToListAsync();
// Generates JOINs - single round-trip but potentially large result set

// Explicit Loading - Load related data on demand
var order = await _context.Orders.FindAsync(id);
await _context.Entry(order)
    .Collection(o => o.Items)
    .LoadAsync();

// Split Queries - avoid cartesian explosion with multiple Includes
var orders = await _context.Orders
    .Include(o => o.Items)
    .Include(o => o.Payments)
    .AsSplitQuery()  // Separate SQL query per Include
    .ToListAsync();

// Projection - BEST for read scenarios
var dtos = await _context.Orders
    .Select(o => new OrderDto
    {
        Id = o.Id,
        Total = o.Items.Sum(i => i.Price * i.Quantity),
        CustomerName = o.Customer.Name
    })
    .ToListAsync();
// Only selects needed columns, no tracking, optimal SQL

Concurrency Handling

// Optimistic concurrency with row versioning
public class Drug
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }

    [Timestamp]
    public byte[] RowVersion { get; set; } // Concurrency token
}

// PostgreSQL uses xmin system column instead of byte[] RowVersion
modelBuilder.Entity<Drug>()
    .UseXminAsConcurrencyToken();

// Handling concurrency conflicts
try
{
    drug.Price = newPrice;
    await _context.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    var entry = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();

    // Strategy: last-write-wins, merge, or reject
    entry.OriginalValues.SetValues(dbValues);
    await _context.SaveChangesAsync(); // Retry with updated token
}
Section 06

SQL - Querying, Indexing, and Isolation Levels

JOINs and Duplicate Rows

-- INNER JOIN: only matching rows from both tables
SELECT d.name, p.amount
FROM drugs d
INNER JOIN prices p ON d.id = p.drug_id;

-- LEFT JOIN: all rows from left + matching from right (NULL if no match)
SELECT d.name, p.amount
FROM drugs d
LEFT JOIN prices p ON d.id = p.drug_id;
-- Includes drugs with no prices (amount = NULL)

-- When JOINs create duplicates:
-- One drug has 3 prices -> drug row appears 3 times in result
-- Fix: use DISTINCT, GROUP BY, or restructure query

-- GROUP BY with HAVING
SELECT pharmacy_id, COUNT(*) as drug_count, AVG(price) as avg_price
FROM drug_prices
GROUP BY pharmacy_id
HAVING COUNT(*) > 10;  -- Filter AFTER aggregation (WHERE filters BEFORE)

EXISTS vs IN | UNION vs UNION ALL

-- EXISTS: stops at first match (better for correlated subqueries)
SELECT d.*
FROM drugs d
WHERE EXISTS (
    SELECT 1 FROM prescriptions p
    WHERE p.drug_id = d.id AND p.date > '2024-01-01'
);

-- IN: evaluates full subquery result set
SELECT * FROM drugs
WHERE id IN (SELECT drug_id FROM prescriptions WHERE date > '2024-01-01');

-- UNION: removes duplicates (sorts + deduplicates = slower)

SELECT name FROM generic_drugs;

SELECT name FROM generic_drugs;

Indexes

-- Clustered Index: determines physical row order (one per table)
-- In PostgreSQL: equivalent is the table itself (heap) + primary key index

-- Non-clustered Index: separate structure pointing to row locations
CREATE INDEX ix_drugs_name ON drugs(name);

-- Covering Index: includes all columns needed by query (no table lookup)
CREATE INDEX ix_drugs_active_price ON drugs(is_active)
    INCLUDE (name, price);
-- Query using only these columns reads ONLY the index

-- Composite Index: column order matters!
CREATE INDEX ix_prices_drug_date ON prices(drug_id, effective_date DESC);
-- Supports: WHERE drug_id = X
-- Supports: WHERE drug_id = X AND effective_date > Y
-- Does NOT efficiently support: WHERE effective_date > Y (missing leftmost)

-- Index trade-off: faster reads, slower writes
-- Every INSERT/UPDATE/DELETE must update all affected indexes

-- 2. Keep transactions short

ACID Principles

  • Atomicity: All operations in a transaction succeed or all fail (rollback)
  • Consistency: Transaction brings DB from one valid state to another (constraints enforced)
  • Isolation: Concurrent transactions don't interfere with each other
  • Durability: Committed transactions survive system crashes (WAL in PostgreSQL)

Circuit Breaker Pattern

States: Closed, Open, Half-Open

Prevents cascading failures. When a downstream service fails repeatedly, the circuit "opens" and fails fast without making calls. After a timeout, it enters "half-open" to test if the service recovered.

    .AddResilienceHandler("pharmacy-pipeline", builder =>
    {
        // Circuit Breaker
        builder.AddCircuitBreaker(new CircuitBreakerStrategyOptions<HttpResponseMessage>
        {
            FailureRatio = 0.5,              // Open after 50% failure rate
            SamplingDuration = TimeSpan.FromSeconds(10),
            MinimumThroughput = 8,           // Need at least 8 calls before evaluating
            BreakDuration = TimeSpan.FromSeconds(30),
            ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
                .HandleResult(r => !r.IsSuccessStatusCode)
                .Handle<HttpRequestException>()
                .Handle<TimeoutRejectedException>()
        });

        // Retry with exponential backoff + jitter
        builder.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay = TimeSpan.FromMilliseconds(200),
            BackoffType = DelayBackoffType.Exponential,
            UseJitter = true,  // Prevents thundering herd
            ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
                .HandleResult(r => r.StatusCode == HttpStatusCode.ServiceUnavailable)
                .HandleResult(r => r.StatusCode == HttpStatusCode.TooManyRequests)
        });

        // Timeout (per-request)

Saga Pattern (Distributed Transactions)

You cannot use ACID transactions across microservices

A Saga is a sequence of local transactions where each step publishes an event triggering the next. If any step fails, compensating transactions undo previous steps. Two approaches: Choreography (event-driven) and Orchestration (central coordinator).

// Orchestration Saga: Prescription Fulfillment
public class PrescriptionFulfillmentSaga
{
    public async Task<SagaResult> ExecuteAsync(PrescriptionRequest request)
    {
        var sagaLog = new SagaLog(request.CorrelationId);

        try
        {
            // Step 1: Verify insurance eligibility
            var eligibility = await _insuranceService.VerifyAsync(request);
            sagaLog.Record("InsuranceVerified", eligibility);

            // Step 2: Reserve inventory at pharmacy
            var reservation = await _inventoryService.ReserveAsync(request.DrugNdc, request.PharmacyId);
            sagaLog.Record("InventoryReserved", reservation);

            // Step 3: Process payment
            var payment = await _paymentService.ChargeAsync(request.PatientId, eligibility.Copay);
            sagaLog.Record("PaymentProcessed", payment);

            // Step 4: Confirm fulfillment
            await _fulfillmentService.ConfirmAsync(reservation.Id);

            return SagaResult.Success();
        }
        catch (Exception ex)
        {
            // Compensate in reverse order
            await CompensateAsync(sagaLog);
            return SagaResult.Failed(ex.Message);
        }
    }

    private async Task CompensateAsync(SagaLog log)
    {
        if (log.Has("PaymentProcessed"))
            await _paymentService.RefundAsync(log.Get<PaymentResult>("PaymentProcessed").TransactionId);
        if (log.Has("InventoryReserved"))
            await _inventoryService.ReleaseAsync(log.Get<ReservationResult>("InventoryReserved").Id);
        // Insurance verification needs no compensation (read-only)
    }
}

Aggregates & Aggregate Roots

Aggregate = Consistency Boundary

An Aggregate is a cluster of domain objects treated as a single unit for data changes. The Aggregate Root is the only entry point. External objects can only reference the root, never internal entities. All invariants are enforced through the root.

// Prescription is the Aggregate Root
public class Prescription  // Aggregate Root
{
    public Guid Id { get; private set; }
    public PatientId PatientId { get; private set; }
    public PrescriptionStatus Status { get; private set; }
    private readonly List<PrescriptionItem> _items = new();
    public IReadOnlyList<PrescriptionItem> Items => _items.AsReadOnly();
    private readonly List<DomainEvent> _domainEvents = new();
    public IReadOnlyList<DomainEvent> DomainEvents => _domainEvents.AsReadOnly();

    // Business logic enforced through Aggregate Root
    public void AddItem(DrugId drugId, Dosage dosage, int quantity)
    {
        if (Status != PrescriptionStatus.Draft)
            throw new DomainException("Cannot modify a submitted prescription");

        if (_items.Count >= 10)
            throw new DomainException("Maximum 10 items per prescription");

        if (_items.Any(i => i.DrugId == drugId))
            throw new DomainException("Drug already in prescription");

        _items.Add(new PrescriptionItem(drugId, dosage, quantity));
        _domainEvents.Add(new ItemAddedToPrescription(Id, drugId));
    }

    public void Submit()
    {
        if (!_items.Any())
            throw new DomainException("Cannot submit empty prescription");
        Status = PrescriptionStatus.Submitted;
        _domainEvents.Add(new PrescriptionSubmitted(Id, PatientId));
    }
}

Bloom Filters - Probabilistic Membership Test

// Bloom Filter: "definitely not in set" or "probably in set"
// Use case: Check if drug interaction pair has been evaluated before
// Saves expensive DB lookup for pairs we've never seen

// Redis module: RedisBloom (BF.ADD, BF.EXISTS)
// Or implement with bit arrays in application code:
public class BloomFilter
{
    private readonly BitArray _bits;
    private readonly int _hashCount;
    private readonly int _size;

    public BloomFilter(int expectedItems, double falsePositiveRate = 0.01)
    {
        _size = (int)(-expectedItems * Math.Log(falsePositiveRate) / (Math.Log(2) * Math.Log(2)));
        _hashCount = (int)(_size / expectedItems * Math.Log(2));
        _bits = new BitArray(_size);
    }

    public void Add(string item)
    {
        foreach (var position in GetHashPositions(item))
            _bits[position] = true;
    }

    public bool MightContain(string item)
    {
        return GetHashPositions(item).All(pos => _bits[pos]);
        // false = DEFINITELY not in set
        // true = PROBABLY in set (check DB to confirm)
    }

    private IEnumerable<int> GetHashPositions(string item)
    {
        var hash1 = item.GetHashCode();
        var hash2 = HashCode.Combine(item, "salt");
        for (int i = 0; i < _hashCount; i++)
            yield return Math.Abs((hash1 + i * hash2) % _size);
    }
}

public async Task<T> GetWithStampedeProtection<T>(string key, Func<Task<T>> factory)
{

    var ttl = await _redis.KeyTimeToLiveAsync(key);

        $"{key}:lock", "1", TimeSpan.FromSeconds(10), When.NotExists);

    if (!lockAcquired && cached.HasValue)
        return JsonSerializer.Deserialize<T>(cached!); // Serve stale

    // Rebuild cache
    var value = await factory();
    await _redis.StringSetAsync(key, JsonSerializer.Serialize(value), TimeSpan.FromMinutes(15));
    await _redis.KeyDeleteAsync($"{key}:lock");
    return value;
}

2. Drug Interaction Checker (Graph - BFS/DFS) Hard

public class DrugInteractionChecker
{
    public static List<List<string>> FindInteractionGroups(
        string[] drugs, string[][] interactions)
    {
        // Build adjacency list
        var graph = new Dictionary<string, HashSet<string>>();
        foreach (var drug in drugs)
            graph[drug] = new HashSet<string>();

        foreach (var pair in interactions)
        {
            if (graph.ContainsKey(pair[0]) && graph.ContainsKey(pair[1]))
            {
                graph[pair[0]].Add(pair[1]);
                graph[pair[1]].Add(pair[0]);
            }
        }

        // BFS to find connected components
        var visited = new HashSet<string>();
        var groups = new List<List<string>>();

        foreach (var drug in drugs)
        {
            if (visited.Contains(drug)) continue;

            var group = new List<string>();
            var queue = new Queue<string>();
            queue.Enqueue(drug);
            visited.Add(drug);

            while (queue.Count > 0)
            {
                var current = queue.Dequeue();
                group.Add(current);

                foreach (var neighbor in graph[current])
                {
                    if (!visited.Contains(neighbor))
                    {
                        visited.Add(neighbor);
                        queue.Enqueue(neighbor);
                    }
                }
            }

            if (group.Count > 1) // Only report groups with interactions
                groups.Add(group);
        }
        return groups;
    }
}
// Time: O(V + E) where V = drugs, E = interactions
// Space: O(V + E) for adjacency list
3. Price History - Maximum Savings Window (Sliding Window) Medium

Solution 
public static decimal MaxSavingsInWindow(decimal[] dailyPrices, int k)
{
    if (dailyPrices.Length < k) return 0;

    // Use a deque to track minimum in sliding window
    var deque = new LinkedList<int>(); // stores indices
    decimal maxSaving = 0;
    decimal averagePrice = dailyPrices.Take(k).Average();

    for (int i = 0; i < dailyPrices.Length; i++)
    {
        // Remove elements outside window
        while (deque.Count > 0 && deque.First.Value <= i - k)
            deque.RemoveFirst();

        // Remove elements larger than current (maintain increasing order)
        while (deque.Count > 0 && dailyPrices[deque.Last.Value] >= dailyPrices[i])
            deque.RemoveLast();

        deque.AddLast(i);

        // Window is full
        if (i >= k - 1)
        {
            decimal windowMin = dailyPrices[deque.First.Value];
            decimal windowMax = dailyPrices.Skip(i - k + 1).Take(k).Max();
            decimal saving = windowMax - windowMin;
            maxSaving = Math.Max(maxSaving, saving);
        }
    }
    return maxSaving;
}
// Time: O(n) - each element added/removed from deque at most once
// Space: O(k) for the deque

public record Pharmacy(int Distance, decimal Price);

public static decimal CheapestReachablePrice(
    Pharmacy[] pharmacies, int maxDistance)
{
    // Sort by distance for binary search
    Array.Sort(pharmacies, (a, b) => a.Distance.CompareTo(b.Distance));

    // Binary search: find rightmost pharmacy within maxDistance
    int left = 0, right = pharmacies.Length - 1;
    int lastReachable = -1;

    while (left <= right)
    {
        int mid = left + (right - left) / 2;
        if (pharmacies[mid].Distance <= maxDistance)
        {
            lastReachable = mid;
            left = mid + 1;
        }
        else
        {
            right = mid - 1;
        }
    }

    if (lastReachable == -1) return -1; // No pharmacy reachable

    // Find minimum price among reachable pharmacies
    decimal minPrice = decimal.MaxValue;
    for (int i = 0; i <= lastReachable; i++)
        minPrice = Math.Min(minPrice, pharmacies[i].Price);

    return minPrice;
}
// Time: O(n log n) sort + O(log n) search + O(n) min-find = O(n log n)

public static string[] TopKDrugs(string[] prescriptions, int k)
{
    // Count frequencies - O(n)
    var freq = new Dictionary<string, int>();
    foreach (var drug in prescriptions)
    {
        freq[drug] = freq.GetValueOrDefault(drug) + 1;
    }

    // Min-heap of size k - keeps top k elements
    var heap = new PriorityQueue<string, int>();

    foreach (var (drug, count) in freq)
    {
        heap.Enqueue(drug, count);
        if (heap.Count > k)
            heap.Dequeue(); // Remove smallest - O(log k)
    }

    // Extract results
    var result = new string[k];
    for (int i = k - 1; i >= 0; i--)
        result[i] = heap.Dequeue();

    return result;
}

6. LRU Cache for Drug Lookups Hard

Implement an LRU (Least Recently Used) cache for frequently queried drug prices. O(1) get and put operations.

Solution 
public class LRUCache<TKey, TValue> where TKey : notnull
{
    private readonly int _capacity;
    private readonly Dictionary<TKey, LinkedListNode<(TKey Key, TValue Value)>> _map;
    private readonly LinkedList<(TKey Key, TValue Value)> _list;

    public LRUCache(int capacity)
    {
        _capacity = capacity;
        _map = new(capacity);
        _list = new();
    }

    public bool TryGet(TKey key, out TValue value)
    {
        if (_map.TryGetValue(key, out var node))
        {
            // Move to front (most recently used)
            _list.Remove(node);
            _list.AddFirst(node);
            value = node.Value.Value;
            return true;
        }
        value = default!;
        return false;
    }

    {

            var lru = _list.Last!;
            _list.RemoveLast();
            _map.Remove(lru.Value.Key);
        }

        var node = _list.AddFirst((key, value));
        _map[key] = node;
    }
}

Given K sorted price lists from different pharmacies, merge them into a single sorted list efficiently.

public static List<DrugPrice> MergeKSortedLists(
    List<DrugPrice>[] lists)
{
    var result = new List<DrugPrice>();

    // Min-heap: (price, listIndex, elementIndex)
    var heap = new PriorityQueue<(int ListIdx, int ElemIdx), decimal>();

    // Add first element from each list
    for (int i = 0; i < lists.Length; i++)
    {
        if (lists[i].Count > 0)
            heap.Enqueue((i, 0), lists[i][0].Price);
    }

    while (heap.Count > 0)
    {
        var (listIdx, elemIdx) = heap.Dequeue();
        result.Add(lists[listIdx][elemIdx]);

        // Add next element from same list
        if (elemIdx + 1 < lists[listIdx].Count)
        {
            heap.Enqueue(
                (listIdx, elemIdx + 1),
                lists[listIdx][elemIdx + 1].Price);
        }
    }

    return result;
}

{

    private readonly TimeSpan _window;
    private readonly Queue<DateTime> _timestamps = new();
    private readonly object _lock = new();

                _timestamps.Enqueue(now);
                return true; // Allowed
            }

            return false; // Rate limited
        }
    }
}

10. Optimal Drug Substitution (Dynamic Programming) Hard

Given a list of drugs with their costs and effectiveness scores, find the combination that maximizes effectiveness within a budget (0/1 Knapsack).

Solution 
public record DrugOption(string Name, int Cost, int Effectiveness);

public static (int MaxEffectiveness, List<string> SelectedDrugs) OptimalCombination(
    DrugOption[] drugs, int budget)
{
    int n = drugs.Length;
    var dp = new int[n + 1, budget + 1];

    // Fill DP table
    for (int i = 1; i <= n; i++)
    {
        for (int w = 0; w <= budget; w++)
        {
            dp[i, w] = dp[i - 1, w]; // Don't take drug i

            if (drugs[i - 1].Cost <= w)
            {
                int withDrug = dp[i - 1, w - drugs[i - 1].Cost]
                    + drugs[i - 1].Effectiveness;
                dp[i, w] = Math.Max(dp[i, w], withDrug);
            }
        }
    }

    // Backtrack to find selected drugs
    var selected = new List<string>();
    int remaining = budget;
    for (int i = n; i > 0; i--)
    {
        if (dp[i, remaining] != dp[i - 1, remaining])
        {
            selected.Add(drugs[i - 1].Name);
            remaining -= drugs[i - 1].Cost;
        }
    }

    return (dp[n, budget], selected);
}
// Time: O(n * budget)
// Space: O(n * budget) - can optimize to O(budget) with 1D array

Implement a Trie that supports prefix-based autocomplete for drug names, returning top-K results by prescription frequency.

public class DrugAutocompleteTrie
{
    private class TrieNode
    {
        public Dictionary<char, TrieNode> Children { get; } = new();
        public bool IsEndOfWord { get; set; }
        public string? FullName { get; set; }
        public int Frequency { get; set; }  // Prescription count for ranking
    }

    private readonly TrieNode _root = new();

    public void Insert(string drugName, int frequency)
    {
        var node = _root;
        foreach (var ch in drugName.ToLowerInvariant())
        {
            if (!node.Children.ContainsKey(ch))
                node.Children[ch] = new TrieNode();
            node = node.Children[ch];
        }
        node.IsEndOfWord = true;
        node.FullName = drugName;
        node.Frequency = frequency;
    }

    public List<string> Autocomplete(string prefix, int topK = 5)
    {
        // Navigate to prefix node
        var node = _root;
        foreach (var ch in prefix.ToLowerInvariant())
        {
            if (!node.Children.TryGetValue(ch, out var child))
                return new List<string>(); // No matches
            node = child;
        }

        // DFS from prefix node, use min-heap for top-K by frequency
        var heap = new PriorityQueue<string, int>();
        CollectWords(node, heap, topK);

        var results = new List<string>();
        while (heap.Count > 0) results.Add(heap.Dequeue());
        results.Reverse(); // Highest frequency first
        return results;
    }

    private void CollectWords(TrieNode node, PriorityQueue<string, int> heap, int k)
    {
        if (node.IsEndOfWord)
        {
            heap.Enqueue(node.FullName!, node.Frequency);
            if (heap.Count > k) heap.Dequeue();
        }
        foreach (var child in node.Children.Values)
            CollectWords(child, heap, k);
    }
}

// Part 2: Minimum rooms/pharmacists needed (sweep line algorithm)
public static int MinRoomsRequired(List<Appointment> appointments)
{
    var events = new List<(DateTime Time, int Delta)>();
    foreach (var apt in appointments)
    {
        events.Add((apt.Start, +1));  // Appointment starts
        events.Add((apt.End, -1));    // Appointment ends
    }

    events.Sort((a, b) => a.Time == b.Time ? a.Delta.CompareTo(b.Delta) : a.Time.CompareTo(b.Time));

    int current = 0, max = 0;
    foreach (var (_, delta) in events)
    {
        current += delta;
        max = Math.Max(max, current);
    }
    return max;
}
// Time: O(n log n), Space: O(n)

public static DateTime? FindNextSlot(
    List<Appointment> booked, TimeSpan duration, DateTime searchFrom)
{
    var sorted = booked.Where(a => a.End > searchFrom).OrderBy(a => a.Start).ToList();

    // Check gap before first appointment
    if (!sorted.Any() || sorted[0].Start - searchFrom >= duration)
        return searchFrom;

    // Check gaps between appointments
    for (int i = 0; i < sorted.Count - 1; i++)
    {
        var gapStart = sorted[i].End;
        var gapEnd = sorted[i + 1].Start;
        if (gapEnd - gapStart >= duration)
            return gapStart;
    }

    // After last appointment
    return sorted.Last().End;
}
3. Consistent Hashing for Cache Distribution Hard

Implement consistent hashing to distribute drug price cache keys across multiple Redis nodes, minimizing cache invalidation when nodes are added/removed.

Solution 
public class ConsistentHashRing<TNode>
{
    private readonly SortedDictionary<int, TNode> _ring = new();
    private readonly int _virtualNodes;  // Replicas per physical node
    private int[] _sortedKeys = Array.Empty<int>();

    public ConsistentHashRing(int virtualNodes = 150)
    {
        _virtualNodes = virtualNodes;
    }

    public void AddNode(TNode node)
    {
        // Add virtual nodes for better distribution
        for (int i = 0; i < _virtualNodes; i++)
        {
            var hash = GetHash($"{node}-vn{i}");
            _ring[hash] = node;
        }
        _sortedKeys = _ring.Keys.ToArray();
    }

    public void RemoveNode(TNode node)
    {
        for (int i = 0; i < _virtualNodes; i++)
        {
            var hash = GetHash($"{node}-vn{i}");
            _ring.Remove(hash);
        }
        _sortedKeys = _ring.Keys.ToArray();
    }

    public TNode GetNode(string key)
    {
        if (_ring.Count == 0) throw new InvalidOperationException("No nodes");

        var hash = GetHash(key);

        // Binary search for first node clockwise from hash
        var idx = Array.BinarySearch(_sortedKeys, hash);
        if (idx < 0) idx = ~idx;  // Bitwise complement = insertion point
        if (idx >= _sortedKeys.Length) idx = 0;  // Wrap around ring

        return _ring[_sortedKeys[idx]];
    }

    private static int GetHash(string key)
    {
        using var md5 = System.Security.Cryptography.MD5.Create();
        var bytes = md5.ComputeHash(Encoding.UTF8.GetBytes(key));
        return BitConverter.ToInt32(bytes, 0);
    }
}

4. Producer-Consumer with Bounded Buffer (Thread-Safe) Hard

Implement a thread-safe bounded buffer supporting multiple producers and consumers with proper synchronization (semaphores, not busy-waiting).

Solution 
public class BoundedBuffer<T> : IDisposable
{
    private readonly T[] _buffer;
    private int _head = 0, _tail = 0, _count = 0;
    private readonly SemaphoreSlim _emptySlots;  // Producers wait when full
    private readonly SemaphoreSlim _fullSlots;   // Consumers wait when empty
    private readonly SemaphoreSlim _mutex = new(1, 1);

    public BoundedBuffer(int capacity)
    {
        _buffer = new T[capacity];
        _emptySlots = new SemaphoreSlim(capacity, capacity);
        _fullSlots = new SemaphoreSlim(0, capacity);
    }

    public async Task ProduceAsync(T item, CancellationToken ct = default)
    {
        await _emptySlots.WaitAsync(ct);  // Block if buffer full
        await _mutex.WaitAsync(ct);
        try
        {
            _buffer[_tail] = item;
            _tail = (_tail + 1) % _buffer.Length;
            _count++;
        }
        finally
        {
            _mutex.Release();
        }
        _fullSlots.Release();  // Signal consumer: item available
    }

    public async Task<T> ConsumeAsync(CancellationToken ct = default)
    {
        await _fullSlots.WaitAsync(ct);   // Block if buffer empty
        await _mutex.WaitAsync(ct);
        T item;
        try
        {
            item = _buffer[_head];
            _buffer[_head] = default!;
            _head = (_head + 1) % _buffer.Length;
            _count--;
        }
        finally
        {
            _mutex.Release();
        }
        _emptySlots.Release();  // Signal producer: slot available
        return item;
    }

    public int Count => Volatile.Read(ref _count);
    public void Dispose()
    {
        _emptySlots.Dispose();
        _fullSlots.Dispose();
        _mutex.Dispose();
    }
}

5. Drug Interaction Network - Shortest Path & Cycle Detection Hard

Model drug interactions as a weighted directed graph where edge weight = severity. Find the shortest severity path between two drugs (indirect interaction chain), and detect circular interaction dependencies.

Solution 
public class DrugInteractionGraph
{
    private readonly Dictionary<string, List<(string Target, int Severity)>> _adj = new();

    public void AddInteraction(string drugA, string drugB, int severity)
    {
        if (!_adj.ContainsKey(drugA)) _adj[drugA] = new();
        if (!_adj.ContainsKey(drugB)) _adj[drugB] = new();
        _adj[drugA].Add((drugB, severity));
        _adj[drugB].Add((drugA, severity));
    }

    // Dijkstra: find minimum-severity interaction path
    public (int TotalSeverity, List<string> Path)? ShortestInteractionPath(
        string source, string target)
    {
        var dist = new Dictionary<string, int>();
        var prev = new Dictionary<string, string?>();
        var pq = new PriorityQueue<string, int>();

        foreach (var node in _adj.Keys)
        {
            dist[node] = int.MaxValue;
            prev[node] = null;
        }

        dist[source] = 0;
        pq.Enqueue(source, 0);

        while (pq.Count > 0)
        {
            var current = pq.Dequeue();
            if (current == target) break;

            if (!_adj.ContainsKey(current)) continue;

            foreach (var (neighbor, weight) in _adj[current])
            {
                var newDist = dist[current] + weight;
                if (newDist < dist[neighbor])
                {
                    dist[neighbor] = newDist;
                    prev[neighbor] = current;
                    pq.Enqueue(neighbor, newDist);
                }
            }
        }

        if (dist[target] == int.MaxValue) return null;

        // Reconstruct path
        var path = new List<string>();
        for (var node = target; node != null; node = prev[node])
            path.Add(node);
        path.Reverse();

        return (dist[target], path);
    }

    // Detect cycles (contraindication loops) using DFS coloring
    public bool HasCycle()
    {
        var state = new Dictionary<string, int>(); // 0=white, 1=gray, 2=black
        foreach (var node in _adj.Keys) state[node] = 0;

        foreach (var node in _adj.Keys)
        {
            if (state[node] == 0 && DfsCycle(node, state, null))
                return true;
        }
        return false;
    }

    private bool DfsCycle(string node, Dictionary<string, int> state, string? parent)
    {
        state[node] = 1; // Gray: currently exploring
        foreach (var (neighbor, _) in _adj[node])
        {
            if (neighbor == parent) continue; // Skip edge we came from
            if (state[neighbor] == 1) return true; // Back edge = cycle
            if (state[neighbor] == 0 && DfsCycle(neighbor, state, node))
                return true;
        }
        state[node] = 2; // Black: fully explored
        return false;
    }
}
// Dijkstra Time: O((V + E) log V)
// Cycle Detection Time: O(V + E)

SOLID Principles

The 5 principles that define clean, maintainable OOP design. You WILL be asked about these.

S - Single Responsibility Principle

A class should have only one reason to change

Each class should do one thing well. If you describe a class with "AND", it probably violates SRP.

        _db.SaveChanges();

        // Notifies

        // Logs

{

    {

    public bool Applies(PatientContext ctx) => ctx.Age >= 65;

public class PriceEngine

        return applicable.Any()
            ? applicable.Min(s => s.Calculate(basePrice))

Mediator Pattern (MediatR)

// Pipeline behaviors for cross-cutting concerns
public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        var failures = _validators
            .Select(v => v.Validate(request))
            .SelectMany(r => r.Errors)
            .Where(f => f != null)
            .ToList();

        if (failures.Any())
            throw new ValidationException(failures);

        return await next(); // Continue pipeline
    }
}

Structured Logging

The #1 Logging Mistake

Using string interpolation $"Processing {orderId}" defeats structured logging. The message template must use named placeholders so log aggregators can index and query by field.

// BAD - string interpolation, unstructured
_logger.LogInformation($"Processing order {order.Id} for {order.CustomerId}");
// Creates a unique message per call - impossible to query/aggregate

// GOOD - message template with named parameters
_logger.LogInformation("Processing order {OrderId} for {CustomerId}",
    order.Id, order.CustomerId);
// Structured: OrderId and CustomerId are indexed, searchable fields

// Log levels guide:
// Trace:       Very detailed, only in development
// Debug:       Internal flow details for debugging
// Information: Business events (order placed, user logged in)
// Warning:     Something unexpected but handled (retry, fallback used)
// Error:       Operation failed, needs attention
// Critical:    System is unusable (DB down, out of memory)

function DrugPriceTable({ drugs }: { drugs: Drug[] }) {
    const [search, setSearch] = useState('');
    const [sortField, setSortField] = useState<keyof Drug>('name');
    const [sortDir, setSortDir] = useState<'asc' | 'desc'>('asc');

    // Memoized filtered + sorted results

        const filtered = drugs.filter(d =>
            d.name.toLowerCase().includes(search.toLowerCase()) ||
            d.ndc.includes(search)
        );

        return [...filtered].sort((a, b) => {
            const aVal = a[sortField];
            const bVal = b[sortField];
            const cmp = aVal < bVal ? -1 : aVal > bVal ? 1 : 0;
            return sortDir === 'asc' ? cmp : -cmp;
        });
    }, [drugs, search, sortField, sortDir]);

    const handleSort = useCallback((field: keyof Drug) => {
        setSortDir(prev =>
            sortField === field && prev === 'asc' ? 'desc' : 'asc'
        );
        setSortField(field);
    }, [sortField]);

    return (
        <div>
            <input
                type="text"
                placeholder="Search by name or NDC..."
                value={search}
                onChange={e => setSearch(e.target.value)}
            />
            <table>
                <thead>
                    <tr>
                        <th onClick={() => handleSort('name')}>Name</th>
                        <th onClick={() => handleSort('price')}>Price</th>
                        <th onClick={() => handleSort('category')}>Category</th>
                    </tr>
                </thead>

                    {displayDrugs.map(drug => (

                            <td>{drug.name}</td>

function AddDrugForm({ onSubmit }: { onSubmit: (drug: Drug) => void }) {
    const [form, setForm] = useState<DrugForm>({
        name: '', ndc: '', price: ''
    });
    const [errors, setErrors] = useState<FormErrors>({});
    const [submitting, setSubmitting] = useState(false);

    const validate = (): FormErrors => {
        const errs: FormErrors = {};
        if (!form.name.trim()) errs.name = 'Name is required';
        if (!/^\d{11}$/.test(form.ndc)) errs.ndc = 'NDC must be 11 digits';
        if (!form.price || Number(form.price) <= 0)
            errs.price = 'Price must be positive';
        return errs;
    };

    const handleSubmit = async (e: React.FormEvent) => {
        e.preventDefault();
        const errs = validate();
        setErrors(errs);
        if (Object.keys(errs).length > 0) return;

        setSubmitting(true);
        try {
            const res = await fetch('/api/drugs', {
                method: 'POST',
                headers: { 'Content-Type': 'application/json' },
                body: JSON.stringify({
                    ...form,
                    price: Number(form.price)
                })
            });
            if (!res.ok) throw new Error('Failed to save');
            const saved = await res.json();
            onSubmit(saved);
            setForm({ name: '', ndc: '', price: '' }); // Reset
        } catch (err) {
            setErrors({ name: err.message });
        } finally {
            setSubmitting(false);
        }
    };

    return (
        <form onSubmit={handleSubmit}>
            <div>
                <input
                    value={form.name}
                    onChange={e => setForm(p => ({...p, name: e.target.value}))}
                    placeholder="Drug name"
                />
                {errors.name && <span className="error">{errors.name}</span>}
            </div>
            <button type="submit" disabled={submitting}>
                {submitting ? 'Saving...' : 'Add Drug'}
            </button>
        </form>
    );
}

function reducer(state: State, action: Action): State {

        case 'FETCH_START':

React Performance & Common Mistakes

Mistakes that show during pair programming
  • Missing key prop: Always use a stable unique ID, NEVER array index (breaks on reorder/delete)
  • Creating objects/arrays in JSX: style={{color: 'red'}} creates new object every render - use useMemo or extract
  • Fetching in useEffect without cleanup: Race conditions when component unmounts or deps change
  • Mutating state directly: state.push(item) doesn't trigger re-render, use setState([...state, item])
  • useEffect with object deps: Objects recreated each render = infinite loop. Destructure primitives or use useMemo

import { useState, useEffect } from 'react';

interface UseFetchResult<T> {
    data: T | null;
    loading: boolean;
    error: Error | null;
    refetch: () => void;
}

export function useFetch<T>(url: string | null): UseFetchResult<T> {
    const [data, setData] = useState<T | null>(null);
    const [loading, setLoading] = useState(false);
    const [error, setError] = useState<Error | null>(null);
    const [trigger, setTrigger] = useState(0);

    useEffect(() => {
        if (!url) return;

        let cancelled = false;
        const controller = new AbortController();
        setLoading(true);

        fetch(url, { signal: controller.signal })
            .then(res => {
                if (!res.ok) throw new Error(`HTTP ${res.status}`);
                return res.json() as Promise<T>;
            })
            .then(data => { if (!cancelled) { setData(data); setError(null); } })
            .catch(err => {
                if (!cancelled && err.name !== 'AbortError') setError(err);
            })
            .finally(() => { if (!cancelled) setLoading(false); });

        return () => { cancelled = true; controller.abort(); };
    }, [url, trigger]);

    const refetch = () => setTrigger(t => t + 1);

    return { data, loading, error, refetch };
}

Async/Await Quiz

Q1: What happens when you await a Task that has already completed?

  • A) It queues the continuation on the thread pool
  • B) It returns synchronously without yielding
  • C) It throws an InvalidOperationException
  • D) It creates a new Task wrapper
If the Task is already complete when awaited, the code continues synchronously - no state machine transition or thread switch occurs. This is why ValueTask exists: to avoid allocating a Task object for the common completed-synchronously path.

Q2: Which of these is safe to do with async void?

  • A) Use it for fire-and-forget background work
  • B) Use it only for event handlers
  • C) Use it whenever you don't need the return value
  • D) Use it in middleware pipeline
async void methods cannot be awaited, and their exceptions propagate to the SynchronizationContext (potentially crashing the process). Only use for event handler signatures that require void return type.

Q3: What does ConfigureAwait(false) actually do?

  • A) Makes the operation run faster
  • B) Prevents the Task from throwing exceptions
  • C) Tells the awaiter not to capture and resume on the original SynchronizationContext
  • D) Cancels the task if it takes too long
It avoids marshaling the continuation back to the original context (e.g., UI thread or ASP.NET request context). In library code, this prevents deadlocks and improves performance. In ASP.NET Core (which has no SynchronizationContext), it has no effect but is still good practice for library code that may be used elsewhere.

EF Core Quiz

Q4: What is the default tracking behavior of a LINQ query in EF Core?

  • A) Tracking - entities are cached in the change tracker
  • B) NoTracking - entities are not tracked
  • C) It depends on the query type
  • D) Tracking is disabled in .NET 8
By default, EF Core tracks all queried entities. Use .AsNoTracking() for read-only scenarios to reduce memory and improve performance. You can change the default with context.ChangeTracker.QueryTrackingBehavior = QueryTrackingBehavior.NoTracking;

Q5: What happens when you call .ToList() in the middle of an IQueryable chain?

  • A) Nothing special, it continues building the expression tree
  • B) It executes the SQL query up to that point and subsequent operations run in-memory
  • C) It throws a compile-time error
  • D) It optimizes the query by adding a batch boundary
ToList() materializes the query - all data up to that point is loaded from the database into memory. Any subsequent .Where(), .Select(), etc. will use LINQ-to-Objects (in-memory), potentially loading much more data than needed. This is a common performance killer.

GC & Memory Quiz

Q6: Objects larger than what size go directly to the Large Object Heap?

  • A) 1 KB
  • B) 64 KB
  • C) 85,000 bytes (~85 KB)
  • D) 1 MB
Objects >= 85,000 bytes are allocated on the LOH. The LOH is collected with Gen 2 (expensive) and is NOT compacted by default, leading to fragmentation. Use ArrayPool<T> to rent large arrays instead of allocating new ones.

Q7: Which lifetime mismatch causes the "captive dependency" bug?

  • A) Transient injected into Scoped
  • B) Scoped injected into Singleton
  • C) Singleton injected into Transient
  • D) Transient injected into Singleton
When a Scoped service (like DbContext) is injected into a Singleton, it gets "captured" - it lives forever instead of being disposed at the end of the HTTP request scope. This means one DbContext shared across ALL concurrent requests: stale data, thread-safety bugs, connection pool exhaustion.

SQL Quiz

Q8: What is the difference between WHERE and HAVING?

  • A) They are interchangeable
  • B) WHERE filters rows before grouping; HAVING filters groups after aggregation
  • C) HAVING is faster than WHERE
  • D) WHERE works on indexed columns; HAVING works on all columns
WHERE filters individual rows before GROUP BY executes. HAVING filters the results of aggregate functions (COUNT, SUM, AVG) after grouping. Example: WHERE price > 10 vs HAVING COUNT(*) > 5

Q9: When does EXISTS outperform IN?

  • A) Always
  • B) Never, IN is always faster
  • C) When the subquery returns many rows (EXISTS short-circuits at first match)
  • D) Only with non-indexed columns
EXISTS stops evaluating as soon as it finds the first matching row (short-circuits). IN must evaluate the entire subquery result set. For large subquery results, EXISTS is typically faster. However, modern query optimizers often rewrite them identically - check the execution plan.

Bug #1: The Race Condition Concurrency 
private static readonly ConcurrentDictionary<string, int> _cache = new();

public int GetOrCompute(string key)
{
    if (!_cache.ContainsKey(key))
    {
        var value = ExpensiveComputation(key);
        _cache[key] = value;
    }
    return _cache[key];
}
Reveal Bug & Fix

Bug: TOCTOU (Time-of-Check-to-Time-of-Use) race condition. Between ContainsKey and the assignment, another thread could have added the key. Also, ExpensiveComputation might run multiple times concurrently for the same key.

// FIX: Use GetOrAdd - atomic operation
public int GetOrCompute(string key)
{
    return _cache.GetOrAdd(key, k => ExpensiveComputation(k));
}

// If computation is expensive and you need EXACTLY once:
private static readonly ConcurrentDictionary<string, Lazy<int>> _cache = new();

public int GetOrCompute(string key)
{
    return _cache.GetOrAdd(key,
        k => new Lazy<int>(() => ExpensiveComputation(k))).Value;
}

public class PriceMonitor
{
    private readonly PriceService _service;

    public PriceMonitor(PriceService service)
    {
        _service = service;
        _service.PriceChanged += OnPriceChanged;

// FIX: Implement IDisposable and unsubscribe
public class PriceMonitor : IDisposable
{

    public PriceMonitor(PriceService service)
    {
        _service = service;
        _service.PriceChanged += OnPriceChanged;
    }

    public void Dispose()
    {
        _service.PriceChanged -= OnPriceChanged; // Unsubscribe!
    }
}

Behavioral Interview Preparation

Often 30-40% of interview weight. Use the STAR method for every answer.

The STAR Method

S Situation - Set the scene in 1-2 sentences. What was the context?

T Task - What was YOUR specific responsibility? (Use "I", not "we")

A Action - What did YOU do? Be specific about technical decisions.

R Result - Quantifiable outcome + what you learned.

Must-Prepare Questions

1. "Tell me about a time you disagreed with an architectural decision."

What they're testing: Technical leadership, communication, ability to commit after debate.

Framework for your answer:

  • What was the decision and why you disagreed (show technical depth)
  • How you presented your alternative (data, prototypes, trade-off analysis)
  • The outcome - even if you "lost", show you committed and made it work
  • What you learned about when that approach IS appropriate
Tip

Never badmouth the person who made the decision. Frame it as "different technical perspectives" not "they were wrong."

2. "Describe a production incident you handled."

What they're testing: Calm under pressure, systematic debugging, root cause analysis.

Framework:

  • How you detected it (monitoring, alerts, customer report?)
  • Your immediate triage steps (severity assessment, communication, mitigation)
  • How you identified root cause (logs, metrics, reproduction)
  • The fix AND what you did to prevent recurrence (tests, monitoring, process change)
3. "Tell me about a time you refactored a large, messy codebase."

What they're testing: Technical judgment, incremental approach, testing strategy.

Key points to hit:

  • Why you chose to refactor NOW (business trigger, not just "code was ugly")
  • Your strategy: incremental vs. big-bang (prefer incremental)
  • How you ensured correctness: added tests BEFORE refactoring (characterization tests)
  • How you communicated risk and progress to stakeholders
4. "How do you handle a junior developer's PR with significant issues?"

What they're testing: Mentoring ability, empathy, constructive communication.

Good answer includes:

  • Start with what's good (acknowledge effort and correct parts)
  • Explain the WHY behind your feedback, not just "change this"
  • Distinguish blocking issues from style preferences
  • Offer to pair on complex changes rather than just requesting changes
  • Follow up to ensure they learned (not just fixed)
5. "Tell me about a time you delivered under a tight deadline. What trade-offs did you make?"

What they're testing: Pragmatism, technical debt awareness, stakeholder communication.

Key points:

  • How you scoped down (what was cut and why)
  • What shortcuts you took and how you documented the tech debt
  • How you communicated scope changes to the team/PM
  • Did you actually go back and pay down the debt? (shows follow-through)
6. "How do you approach estimating work for complex features?"

What they're testing: Planning, risk awareness, intellectual honesty.

Good answer:

  • Break into smaller tasks (spike if needed for unknowns)
  • Identify risks and unknowns upfront (integrations, data migration, etc.)
  • Add buffer for testing, code review, unexpected issues (typically 1.5-2x)
  • Communicate uncertainty ranges, not single numbers
  • Re-estimate as you learn more, communicate changes early
7. "Describe how you ensure data accuracy in healthcare systems."

Healthcare-specific - shows domain awareness:

  • Validation at every boundary (API input, file import, inter-service)
  • Idempotent operations to handle retries safely
  • Audit trails: who changed what, when (immutable event log)
  • Reconciliation jobs to catch drift between systems
  • Alerting on anomalies (sudden price changes, missing data)
  • Testing with production-like data volumes

How to Structure Your Interview Answers

The 3-Layer Answer Framework (for technical questions)
  1. Definition (10 seconds) - Show you know the concept
  2. How it works internally (30-60 seconds) - Show depth
  3. Trade-offs and when to use/avoid (30 seconds) - Show judgment
  4. Real example from experience (30 seconds) - Show it's not just theory

Interviewers promote candidates who show JUDGMENT, not just knowledge. Always end with trade-offs.

Red Flags - Answers That Disqualify Immediately

Never say these in an interview
  • "I always use .Result to get the value from async methods"
  • "I don't write tests - I test manually"
  • "I just catch(Exception) and log it everywhere"
  • "I don't know what the GC does, it just works"
  • "I use the same DbContext as a singleton for the whole app"
  • "I just use SELECT * in all my queries"
  • "I've never needed to think about performance"

What is the difference between Task.WhenAll and Task.WhenAny?
WhenAll completes when ALL tasks finish (returns all results). WhenAny completes when the FIRST task finishes (returns that single task). Use WhenAny for timeout patterns or first-response-wins.
What causes LOH fragmentation and how do you fix it?
Objects >= 85KB go to LOH. Not compacted by default. Allocating/freeing large objects leaves holes. Fix: ArrayPool<T> to rent arrays, avoid large short-lived allocations, LOH compaction mode sparingly.
What is the difference between AddDbContext and AddDbContextPool?
AddDbContextPool reuses DbContext instances (change tracker cleared between uses). Better performance but you cannot inject scoped services into the context constructor. Use for high-throughput read scenarios.
Explain the Outbox Pattern in one sentence.
Save the domain event to a local "outbox" table in the SAME transaction as the business operation, then a background process publishes those events to the message broker - guaranteeing at-least-once delivery without distributed transactions.
When should you use ValueTask<T> instead of Task<T>?
When the method completes synchronously MOST of the time (e.g., cache hits). ValueTask avoids the heap allocation of a Task object. DANGER: Never await a ValueTask twice, never use .Result before completion, never cache a ValueTask.
What is the difference between IOptions, IOptionsSnapshot, and IOptionsMonitor?
IOptions<T>: singleton, read once at startup. IOptionsSnapshot<T>: scoped, re-reads config per request (good for hot-reload). IOptionsMonitor<T>: singleton, notifies on change via OnChange callback.
